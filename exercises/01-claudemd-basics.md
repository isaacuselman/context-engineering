# Exercise 01 — CLAUDE.md basics

## Why this matters

CLAUDE.md is the highest-leverage file in the Claude Code harness: it is injected at the start of every session in the current project, wrapped in a `<system-reminder>` tag, and it compounds across every turn. A good one makes every future session sharper. A bad one — auto-generated by `/init` and shipped unedited, stuffed with style rules that should live in a linter, or 500 lines of caps-lock threats — degrades adherence to *all* your instructions, not just the bad ones.

The goal of this exercise is to write a CLAUDE.md for this very repo (`phase-0/`) and then critique it against the rules distilled from HumanLayer's guide, Anthropic's best-practices post, and Boris Cherny's self-improvement loop.

## Concepts

**Instruction budget.** Frontier thinking LLMs reliably follow around 150–200 instructions (HumanLayer, citing arXiv 2507.11538). Claude Code's own system prompt already spends ~50 of those. Every line you add eats into the remainder — and each added line degrades adherence to *all* the rules, not just the newest one. Write less on purpose.

**Universal applicability.** Claude Code wraps your CLAUDE.md in a `<system-reminder>` that explicitly tells the model "this context may or may not be relevant…you should not respond to this context unless it is highly relevant." Irrelevant content is double-penalized: ignored AND pollutes context. If a rule only applies when you're modifying tests, wrap it in a conditional block (`<important if="you are writing or modifying tests">…</important>`). Don't wrap foundational items (tech stack, directory map).

**Pointers beat snippets.** Embedded code goes stale the moment the file changes. Use `path/to/file.ts:42` pointers instead. Claude can open the file if it needs the details.

**Don't make Claude be a linter.** Formatting and naming rules are what Biome, Prettier, and `Stop` hooks enforce deterministically. LLMs already absorb style from existing code via in-context learning. Spending instruction budget on "use tabs not spaces" is a waste.

**The self-improvement loop.** Cherny's canonical rule: "anytime we see Claude do something incorrectly, we add it to CLAUDE.md so it doesn't repeat next time." Operationally — when Claude makes a mistake, prompt it with `Update CLAUDE.md so you don't make that mistake again`. Check it into git. Update it multiple times a week.

**Anti-patterns to avoid** (§1.5):

- Shipping `/init` output unedited
- Embedded code snippets
- Style/formatting rules
- Caps-lock threats on every line
- Settings-level config in prose ("NEVER add Co-Authored-By…") — that belongs in `.claude/settings.json`
- 500+ line files

## Exercise

**Write a CLAUDE.md for this repo (`phase-0/`) at no more than 60 lines.**

Suggested structure (mirroring `claude-code-action/CLAUDE.md`):

```markdown
## Commands
## What This Is
## How It Runs
## Key Concepts
## Things That Will Bite You
## Code Conventions
```

Concrete content hints drawn from this repo:

- **Commands:** how to serve the app locally (`python3 -m http.server 8000`), why `file://` breaks it
- **What this is:** a single-page HTML app that renders markdown exercises and tracks progress in localStorage
- **How it runs:** one `<script>` block in `index.html`; `render()` reads the hash, fetches `exercises/{slug}.md`, runs `marked.parse()`, appends the textarea and button
- **Key concepts:** the `EXERCISES` array is the source of truth (no manifest file); localStorage keys are `completed-{id}` and `notes-{id}`; marked.js is pinned to `@14`
- **Things that will bite you:** `file://` fetches blocked; localStorage is origin-scoped so `localhost:8000` and GitHub Pages are different stores; Charter font is not installed by default so Georgia is the real body font on most systems
- **Code conventions:** plain HTML/CSS/JS only, no build step, single `<style>` and single `<script>` block

Save your draft as `CLAUDE.md` in the repo root. Then run the critique below.

### Self-critique checklist

Go through every line of your CLAUDE.md and answer yes/no to each:

1. Is this instruction **universally applicable** in this repo, or only relevant sometimes? If sometimes, wrap it in a conditional.
2. Is this a **linter's job** (style, formatting, naming)? If yes, delete it.
3. Is this **settings-level config** that should live in `.claude/settings.json`? If yes, move it.
4. Does this line contain an **embedded code snippet** that could go stale? If yes, replace with a `path/line` pointer.
5. Would a smart new contributor to this repo want to know this on day one? If no, delete it.
6. Total line count ≤ 60? If not, what can you cut?
